1. HOOK — THE PROBLEM WORTH SOLVING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The Payment Selection Page (PSP) is the single most critical screen in
Amazon India's checkout — every rupee of GMV flows through it. Yet there
was no single source of truth for how it should look, behave, or be built.

The reality:
• 90+ touchpoints rendering PSP across Amazon Shopping and Paycheckout
• 2 separate design libraries (RIO/Stores Design Library + Tuxedo Mobile)
• 2 engineering cohorts building the same screen with different interpretations
• Designers redrawing instrument tiles from scratch for every project
• Engineers guessing at spacing, radio button sizing, and selection behavior
• No documented rules for preselection logic, badge priority, or section order
• Inconsistencies: 4 dots vs. 2 dots in card numbers, different badge colors,
  different radio button sizes, inconsistent border-radius on selection

The cost: every inconsistency erodes user trust at the moment of payment.
When a customer hesitates for even 2 seconds because something "looks off,"
conversion drops. At Amazon scale, that's measurable revenue.

(LP: Customer Obsession — starting from what breaks the customer experience,
not from what's convenient for the design team)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Amazon India checkout has two rendering contexts for the same PSP:

┌─────────────────────────┬───────────────────────────────────┐
│ Paycheckout Context     │ Stores Checkout Context           │
├─────────────────────────┼───────────────────────────────────┤
│ WebView inside app      │ Native Stores checkout            │
│ Tuxedo Mobile library   │ RIO Foundations library           │
│ Tuxedo handles buttons, │ Child: Stores Design Library      │
│ inputs, typography      │ RIO handles buttons, inputs, type │
│ Single engineering team  │ Multiple Stores teams             │
└─────────────────────────┴───────────────────────────────────┘

Both contexts render the SAME PSP instrument tiles — same icons, badges,
radio buttons, section hierarchy. But:
• Designers had Figma files with no documented spec for tile states
• Engineers had working code but no pixel-level reference
• Product managers had requirements but no visual decision framework
• New team members needed weeks to understand preselection logic
• Nobody documented WHY certain decisions existed

The gap was between the two libraries: a "PSP layer" that sits on top of
both Tuxedo and RIO, defining payment-specific behavior that neither
parent library covers.

Principle: if the documentation IS the product, it stays maintained.
Not a static PDF. Not a Figma-only spec. A living website.

Constraints I set:
• Static site (GitHub Pages — zero backend, zero infrastructure)
• No build tools (vanilla HTML/CSS/JS — open index.html, it works)
• Usable by designers, engineers, AND product managers
• Shows real pixels and interactions, not descriptions of them
• Documents decisions (WHY), not just outputs (WHAT)

I used Kiro AI as a development partner throughout this project. My role:

execution. This wasn't about replacing thinking — it was about compressing
a quarter of work into weeks while maintaining high standards.

4. KEY DESIGN DECISIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DECISION 1: Semi-Modernized PSP — Radio Buttons, Not Full-Tile Tap
──────────────────────────────────────────────────────────────────
Context: Modern payment UIs (GPay, PhonePe) use full-tile tap selection.
Should we follow?

Decision: No. Radio button selection with modern visual styling.

Why:
• PSP tiles contain clickable sub-elements: "Details" offer link,
  bank pill selector (change UPI bank), "Add Now" CTA for insufficient
  balance. Full-tile tap creates ambiguity.
• Radio provides unambiguous 44×44px selection target
• Indian users are deeply familiar with radio-based forms (bank forms,
  government portals, e-commerce)
• We get modern aesthetics (blue border, subtle bg highlight) WITH
  traditional interaction clarity

The selected state: bg #EDF8FF, border 2px solid #2162A1, badge turns
blue (#0A7CD1), name goes bold (700), radio fills with blue dot.

(LP: Customer Obsession — familiarity builds trust at payment moment)


DECISION 2: Position-Aware Border Radius
─────────────────────────────────────────
Context: When a tile is selected inside a card group, what border-radius
should the selection highlight have?

Problem discovered: giving all selected tiles border-radius:12px created
visual artifacts — the selection looked "floating" inside the group rather
than being part of it.

Solution: Position-aware radius lookup:
• Top tile: 12px 12px 0 0 (matches group top corners)
• Middle tile: 0 (flush with group edges)
• Bottom tile: 0 0 12px 12px (matches group bottom corners)
• Only tile: 12px (full radius, matches entire group)

Implementation: data-attribute based tile indexing (data-tile="0",
data-tile="1") with parent group analysis at init time. NOT CSS nth-child
because tiles can be dynamically added/removed.

(LP: Insist on the Highest Standards — pixel-level correctness matters
when building a design system that others will reference)


DECISION 3: Badge Priority System
──────────────────────────────────
RECOMMENDED section max: 3 instruments. But which 3? And in what order?

Badge priority defines both sorting AND preselection:
1. Best offer — highest savings (strongest conversion signal)
2. Previously used — last successful payment in 30 days (trust)
3. Featured — system-promoted (weakest signal)

The first tile is ALWAYS auto-selected on page load.
Why Best Offer first, not Previously Used?
• Savings is a quantifiable value proposition (₹50 saved)
• Previously used is implicit trust (no visible benefit)
• A/B testing showed Best Offer preselection has higher continue-rate

(LP: Are Right, A Lot — data-driven prioritization over gut feeling)


DECISION 4: Type-Based Section Routing (Absolute Rule)
──────────────────────────────────────────────────────
• UPI-type → UPI section. ALWAYS. Even without a badge.
• Card-type → CARDS section. ALWAYS.
• Badge determines if instrument ALSO appears in RECOMMENDED.
• Everything else → MORE WAYS TO PAY.

This seems obvious but was being violated in production: some flows put
Amazon Pay UPI in "More Ways to Pay" when it didn't have a badge. We
documented this as an absolute rule — type determines native section,
badge determines promotion to RECOMMENDED.

(LP: Have Backbone; Disagree and Commit — establishing clear rules even
when existing implementations contradict them)


DECISION 5: LLM Over Regex for AI Generator (The Pivot)
───────────────────────────────────────────────────────
This was the most important technical decision of the project.

What we tried first: A 600-line rule-based "AI" parser using keyword
extraction, proximity matching, comma-splitting, and regex patterns.

What happened: It broke on every novel prompt. Over 3 weeks we rewrote
the badge parser 4 times, the offer extractor 3 times, and the section
routing twice. Each fix created new edge cases:
• "Featured" duplicating across multiple instruments
• ₹60,000 credit limit being parsed as order amount
• "APL in Featured" misinterpreted (section vs. badge)
• UPI instruments not routing to UPI section

Root cause diagnosis: NLP with regex is structurally incapable of handling
ambiguous natural language from diverse users. No amount of tuning fixes
a category error.

The decision: Delete 550 lines of regex code. Replace with a 50-line API
call to OpenRouter (free tier LLM). A comprehensive system prompt teaches
the LLM all PSP rules — badges, section routing, instrument states,
preselection logic.

Result: Works perfectly on every prompt we've tested. Zero code changes
needed for new prompt patterns. The LLM understands intent the way a
human does.

(LP: Invent and Simplify — sometimes the simplest solution is removing
code, not adding it. Also: Learn and Be Curious — recognizing when to
adopt a fundamentally different approach)

6. CHALLENGES & HOW I SOLVED THEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CHALLENGE 1: Mental Model Interactivity (3 rewrites)
────────────────────────────────────────────────────
The Mental Model PSP needed click-to-select with exact state transitions.
But the static HTML had no consistent structure — tiles were written inline
with varied patterns.

Attempt 1: querySelector by style properties → fragile, broke on minor CSS changes
Attempt 2: nth-child targeting → broke when tiles were reordered
Attempt 3: data-attribute system (data-tile, data-name, data-badge) + init-time
           structural analysis → stable, works regardless of HTML changes

The data-attribute approach required analyzing tile patterns at page load:
Pattern A (RECOMMENDED): padding:12px wrapper → child flex div → SVG
Pattern B (other sections): padding:12px + display:flex combined → SVG

Both patterns detected and tagged automatically.


CHALLENGE 2: Radio Button SVG Manipulation
─────────────────────────────────────────
Problem: Radio buttons are SVG circles. Selection requires:
• Changing outer circle stroke from #D5D9D9 to #2162A1
• CREATING an inner filled circle (r=5, fill=#2162A1)
• On deselection: REMOVING the inner circle, resetting stroke

This requires DOM manipulation of SVG elements (createElementNS, not
createElement). Multiple bugs from:
• Using wrong namespace (HTML vs. SVG)
• Not cleaning up inner circles on deselect (accumulating fills)
• Circle count check (querySelectorAll('circle')[1]) breaking when
  inner circle didn't exist

Fix: Explicit create/remove cycle with namespace-aware SVG creation.


CHALLENGE 3: Flex Alignment + Insufficient Balance CTA
─────────────────────────────────────────────────────
Problem: Amazon Pay Balance tile has "Add ₹X to proceed | Add Now ›"
below the name. Using align-items:center on the flex row pushed the
radio button and icon down, misaligning them with the instrument name.

First fix: Changed to align-items:flex-start globally → broke simple
tiles (just name, no details) where icon/radio should be centered.

Final fix: Conditional alignment — detect whether tile has extra content
(badge, details, offer, bankPill, insufficientCta). If yes: flex-start
with margin-top:2px on icon and radio. If no: center.


CHALLENGE 4: Dot Count Standardization
─────────────────────────────────────
Audit found: card details had "••••0424" (4 dots) in some places and
"••0424" (2 dots) in others. Inconsistent across:
• instrument-registry.js (data definitions)
• app.js Mental Model static HTML
• Component preview examples
• UPI bank pill text
• Code example snippets

Fix: Global find-replace to 2 dots (••) with verification grep confirming
zero 4-dot instances remain. Five separate locations fixed.


CHALLENGE 5: Indian Number System
────────────────────────────────
Problem: Amounts like "59940" displaying as "59,940" (Western grouping)
when Indian users expect "59,940" (same in this case) but for larger
numbers: "1,00,000" not "100,000".

Fix: formatIndian() utility — splits into last-3 (thousands) and applies
2-digit grouping to remainder. Applied consistently in CTA price, savings
bar, and insufficient balance shortfall.


CHALLENGE 6: Dark Mode Failed WCAG AA
─────────────────────────────────────
Implemented full dark mode with theme toggle and localStorage persistence.
Post-implementation audit revealed multiple contrast failures:
• Secondary text (#565959 on dark bg) failed 4.5:1 minimum
• Badge backgrounds didn't meet 3:1 for UI elements
• Savings bar green lost readability on dark surface

Decision: Disabled dark mode entirely (init function is a no-op) rather
than ship an inaccessible version. Will re-enable when a proper dark
palette is designed and verified. The toggle code remains for when ready.

(LP: Insist on the Highest Standards — inaccessible is worse than absent)


CHALLENGE 7: The Regex Parser Was Fundamentally Wrong (3 weeks of fixes)
───────────────────────────────────────────────────────────────────────
Documented extensively in Decision 5 above. The key learning: when you're
on your 4th rewrite of the same component and it still breaks on new
inputs, the problem isn't the implementation — it's the approach.

Weeks spent: badge parser rewrite (×4), offer extraction fix (×3),
section routing fix (×2), order amount extraction fix (×2).

The permanent fix took 1 hour: delete the parser, write a system prompt,
wire an API call. 550 lines removed, 50 added.

8. WHAT I'D DO DIFFERENTLY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Start with LLM integration from day one for the AI generator. The 3 weeks
   spent iterating on regex parsing was a sunk cost that should have been
   avoided. The signal was clear after the second rewrite — if the problem
   is "understand human intent," use a system designed for that, not regexes.

2. Document both rendering contexts (Paycheckout + Stores) from the start.
   I built for the semi-modernized context first, planning to add Stores
   later. But the two contexts share 90% of PSP behavior — capturing both
   early would have forced cleaner architectural separation between the
   "PSP layer" and the "chrome layer."

3. Set up automated visual regression tests earlier. Several bugs (dot
   count, radio alignment, badge colors) were caught by manual inspection
   after shipping. A snapshot comparison system would have caught these
   before they went live.

(LP: Learn and Be Curious — every project teaches where to start next time)
